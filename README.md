# Orrel: eBPF VM and JIT-compiler

This is a scratchpad project where I will build, from scratch and in Rust, a complete eBPF execution system with no eBPF virtual-machine or JIT crate underneath it. The finished system has four interlocking components, and they are built in this order because each one earns the right to the next.

The first component is an interpreter. It decodes the eBPF instruction stream directly and executes the full instruction set, so that I can prove I completely understand what every instruction means before I ever try to compile one. To ensure comprehensive coverage of the base specification, I am fully implementing the standard eBPF 64-bit register set, memory model, and Arithmetic Logic Unit (ALU) operations.

The second component is a just-in-time (JIT) compiler that emits native RISC-V 64-bit (RV64) machine code. It generates RV64 instructions by hand into an executable memory mapping and runs them. This ensures that I fully own the instruction selection, register allocation, machine-code encoding, the write-xor-execute (W^X) transition (utilizing operating system mechanisms to explicitly manage memory page permissions), and the host calling convention from end to end.

The third component is a verifier. It performs strict control-flow analysis to guarantee program termination, rigorous register-state and type tracking, and memory bounds checking on untrusted bytecode. It immediately rejects any program it cannot mathematically prove safe, ensuring that I internalise the discipline of treating generated bytecode as inherently adversarial until proven otherwise.

The fourth component is a differential-testing harness. It systematically generates and mutates valid eBPF bytecode, runs each generated program through both the interpreter and the JIT compiler, and asserts bit-for-bit equivalence of the full observable machine state, including all registers and stack memory.
