# Orrel: eBPF VM and JIT-interpreter

This is a scratchpad project where I will build, from scratch and in Rust, a complete eBPF execution system with no eBPF virtual-machine or JIT crate underneath it. The finished system has four interlocking components, and they are built in this order because each one earns the right to the next.

The first component is an interpreter. It decodes the eBPF instruction stream by hand and executes the full instruction set, so that the engineer (me) proves he understands what every instruction means before he ever tries to compile one.

The second component is a just-in-time compiler that emits native RISC-V 64-bit machine code. It emits RV64 instructions by hand into an executable memory mapping and runs them, so that the engineer (me) owns instruction selection, register allocation, machine-code encoding, the write-xor-execute transition, and the host calling convention end to end.

The third component is a verifier. It performs control-flow analysis, register-state tracking, and bounds checking on untrusted bytecode and rejects any program it cannot prove safe, so that the engineer (me) internalises the discipline of treating generated bytecode as adversarial until proven otherwise.

The fourth component is a differential-testing harness. It generates and mutates valid bytecode, runs each program through both the interpreter and the JIT, and asserts bit-for-bit equivalence of the full observable machine state, so that the engineer (me) builds the exact oracle whose absence caused his earlier failure.
